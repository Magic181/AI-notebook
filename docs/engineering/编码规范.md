# 编码规范

## 通用原则

- 代码可读性优先
- 避免过度抽象
- 单一职责原则
- DRY原则

## 命名规范

### Python (Backend)

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | snake_case | `user_service.py` |
| 类名 | PascalCase | `UserService` |
| 函数名 | snake_case | `get_user_by_id` |
| 变量名 | snake_case | `user_name` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |

### TypeScript (Frontend)

| 类型 | 规范 | 示例 |
|------|------|------|
| 文件名 | kebab-case | `user-service.ts` |
| 组件文件名 | PascalCase | `UserCard.vue` |
| 类名 | PascalCase | `UserService` |
| 函数名 | camelCase | `getUserById` |
| 变量名 | camelCase | `userName` |
| 常量 | UPPER_SNAKE_CASE | `MAX_RETRY_COUNT` |
| 接口名 | PascalCase | `UserInterface` |

## 代码风格

### Python

- 使用 Black 格式化
- 使用 isort 排序导入
- 行长度：88字符
- 使用类型注解

### TypeScript

- 使用 Prettier 格式化
- 使用 ESLint 检查
- 启用 strict 模式
- 使用 interface 定义类型

## 注释规范

### 必须注释

- 复杂业务逻辑
- 正则表达式
- 临时解决方案（TODO/FIXME/HACK）
- API接口说明

### 无需注释

- 显而易见的代码
- 重复的注释

## Git提交规范

```
<type>(<scope>): <subject>

<body>

<footer>
```

### Type

- feat: 新功能
- fix: 修复bug
- docs: 文档更新
- style: 代码格式
- refactor: 重构
- test: 测试
- chore: 构建/工具

### 示例

```
feat(notebook): 实现Notebook创建功能

- 添加Notebook模型
- 实现CRUD接口
- 添加权限验证

Closes #123
```

## 仓库卫生

- 本地密钥、数据库、上传文件和手测脚本不得提交。
- `.env` 只保存在本地，新增环境变量时同步更新 `.env.example` 和环境配置文档。
- 手动上传测试文件放在 `backend/test_uploads/`，一次性脚本可使用 `backend/test_*.py`，这些文件默认被 Git 忽略。
- 自动化测试应放在对应 app 的 `tests.py` 或专用测试包中，确保可以被 `manage.py test` 发现。
- 提交前使用 `git status --short` 检查暂存范围，只提交与当前批次相关的文件。

## 换行规范

- 文本文件统一使用 LF，规则由 `.gitattributes` 固定，避免 Windows/macOS/Linux 之间产生空 diff。
- Windows 专用脚本（`.bat`、`.cmd`、`.ps1`）保留 CRLF，方便系统原生执行。
- 若 `git status` 显示 modified 但 `git diff` 为空，优先检查 `git ls-files --eol <file>`，不要把行尾变更混入业务提交。
